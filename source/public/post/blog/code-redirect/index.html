代码迁移和维护的一些技巧。

  我们平常说的重定向，是指网址重定向。   比如，当年的京东网址是360buy.com，后来升级成了jd.com。不知道的用户肯定还记着旧的网址，那怎么办呢？我们就把访问360buy.com的用户重定向到jd.com。   在代码维护的过程中，我们可能也会遇到给运行中汽车换轮胎这样棘手的问题。   随着功能的迭代，可能版本1的代码和版本2的代码有了较大的差异，版本1的代码已经有其他程序员在用了，但是版本2的代码又必须上。把版本1的代码打包服务化吧，或者我们推倒重构吧。这样的方式代价有点大。在这个过渡阶段，我们就可以借鉴网址重定向的思路。

类的层面

  项目初期，很多代码都是放在一个目录下的（单体应用）。随着业务的发展，某些基础代码可能要做为服务化做准备了；某些业务可能要分家了。   这里以一个电商的项目为例：   当业务还小的时候订单系统就一个文件。但是当业务规模变大，里面的每个文件都变得非常大，可能有5000行以上的代码。这个时候就需要重构或者拆分代码了。   让原有的Home\DealLogic继承Deal\DealLogic,Home\DealLogic变成了一个空文件，并没有实际的代码。所有的逻辑都迁移到了Deal目录下。通过类的继承，把所有对Home\DealLogic的调用都路由到了Deal\DealLogic。新旧代码之间依然有交互的地方，就让Deal\DealLogic分发到Deal下面的逻辑。

函数的层面

  某个函数版本1和版本2的增量改动比较大，维护的时候怎么办呢？   起两个名字吧，function v1以及function v2。v1继续用，新的业务就用v2嘛。但是如果v2跟v1是向下兼容的呢，而且v2里面有很多代码跟v1是兼容的，那岂不是有很多的冗余代码?

变量的层面

  有时候，我们可能把某个变量的名称拼写错了;或者变量太多了，零零散散，想把它封装成对象;又或者我们想通过getter增加其他的转换或功能等等。但是原本的变量名称已经被很多其他同事在使用了，怎么办呢？   在C++里面有个取别名的功能，新的名字跟旧的名字指向同一个引用。
